# Requirements Document

## Introduction

This document defines the requirements for an AI-powered customer service chatbot built on a local LLM stack. The system provides a web-based chat interface (Angular) through which customers can ask questions and receive contextually accurate answers. Responses are generated by a locally-hosted Llama 3.2 model served via Ollama (at `192.168.1.88`), augmented by a Retrieval-Augmented Generation (RAG) pipeline that grounds answers in an ingested customer service knowledge base stored in a vector database. The system is containerised with Docker and deployable to Kubernetes, with full observability through a monitoring stack.

---

## Glossary

- **Chat_UI**: The Angular web application that presents the chat interface to the end user.
- **API_Gateway**: The backend HTTP service that receives chat requests from the Chat_UI and orchestrates the RAG pipeline and LLM calls.
- **Ollama_Client**: The component within the API_Gateway responsible for communicating with the Ollama inference server.
- **Ollama_Server**: The locally-hosted Ollama instance at `192.168.1.88` running the `llama3.2` model.
- **RAG_Pipeline**: The Retrieval-Augmented Generation pipeline that retrieves relevant knowledge base chunks and injects them into the LLM prompt as context.
- **Embedding_Service**: The component that converts text (queries and documents) into vector embeddings for semantic search.
- **Vector_DB**: The vector database (e.g., Qdrant or Chroma) that stores and indexes document embeddings for similarity search.
- **Ingestion_Pipeline**: The offline process that loads, chunks, embeds, and stores knowledge base documents into the Vector_DB.
- **Knowledge_Base**: The collection of customer service documents (FAQs, policies, product guides) used to ground chatbot responses.
- **Conversation_History**: The ordered sequence of user and assistant messages within a single chat session.
- **Session**: A single continuous interaction between one user and the chatbot, identified by a unique session ID.
- **Turn**: One user message plus one assistant reply pair within a Session.
- **Chunk**: A fixed-size, optionally overlapping segment of a source document produced by the Ingestion_Pipeline.
- **Context_Window**: The maximum number of tokens the Ollama_Server can process in a single inference request.
- **Health_Endpoint**: An HTTP endpoint that reports the operational status of a service, returning HTTP 200 when healthy and HTTP 503 when unhealthy.
- **Metrics_Endpoint**: An HTTP endpoint that exposes Prometheus-compatible metrics for a service.

## Requirements

### Requirement 1: Chat Interface

**User Story:** As a customer, I want a web-based chat interface, so that I can ask questions and receive helpful responses without leaving my browser.

#### Acceptance Criteria

1. THE Chat_UI SHALL render a chat window containing a scrollable message history area and a text input field with a send button.
2. WHEN a customer submits a message, THE Chat_UI SHALL display the message in the history area immediately and send it to the API_Gateway before the API_Gateway responds.
3. WHILE the API_Gateway is processing a response, THE Chat_UI SHALL display a visible loading indicator in the message history area.
4. WHEN the API_Gateway returns any response (including errors), THE Chat_UI SHALL remove the loading indicator and append the response content to the message history area.
5. IF the API_Gateway returns an error, THEN THE Chat_UI SHALL display a user-readable error message as an additional message in the message history area without clearing or modifying the existing Conversation_History.
6. THE Chat_UI SHALL preserve the full Conversation_History for the duration of the browser session.
7. WHEN a customer presses the Enter key in the text input field, THE Chat_UI SHALL submit the message as if the send button were clicked.
8. THE Chat_UI SHALL disable the text input field and send button while a response is pending.
9. THE Chat_UI SHALL support rendering of markdown-formatted text in assistant messages, including bold, italic, bullet lists, and code blocks.
10. THE Chat_UI SHALL provide a "New Conversation" control that clears the Conversation_History and starts a new Session.

---

### Requirement 2: Backend API Service

**User Story:** As a system operator, I want a backend API service, so that the Chat_UI has a stable, versioned interface for sending messages and receiving responses.

#### Acceptance Criteria

1. THE API_Gateway SHALL expose a RESTful HTTP endpoint at `POST /api/v1/chat` that accepts a JSON body containing a session ID, the customer message, and the Conversation_History.
2. WHEN a valid chat request is received and all downstream processing succeeds, THE API_Gateway SHALL return an HTTP 200 response containing the assistant reply as a JSON body within 30 seconds.
3. IF a chat request body is missing required fields or the customer message exceeds 10,000 characters or the Conversation_History contains more than 100 entries, THEN THE API_Gateway SHALL return an HTTP 400 response with a JSON error body describing the specific validation failure.
4. IF the Ollama_Server is unreachable, THEN THE API_Gateway SHALL return an HTTP 503 response with a JSON error body indicating the inference service is unavailable.
5. IF the Vector_DB is unreachable, THEN THE API_Gateway SHALL return an HTTP 503 response with a JSON error body indicating the retrieval service is unavailable.
6. IF all downstream dependencies respond within 5 seconds on a `GET /api/v1/health` probe, THEN THE API_Gateway SHALL return HTTP 200 with a JSON body listing each dependency and its status.
7. IF any downstream dependency does not respond within 5 seconds on a `GET /api/v1/health` probe, THEN THE API_Gateway SHALL return HTTP 503 with a JSON body identifying the unresponsive dependency.
8. THE API_Gateway SHALL expose a `GET /metrics` Metrics_Endpoint in Prometheus exposition format.
9. THE API_Gateway SHALL accept Cross-Origin Resource Sharing (CORS) requests from the configured Chat_UI origin.
10. THE API_Gateway SHALL log each request and response at INFO level, including session ID, request duration in milliseconds, and HTTP status code, without logging the content of customer messages.
11. WHEN a request exceeds 30 seconds, THE API_Gateway SHALL cancel the downstream inference call and return an HTTP 504 response.

### Requirement 3: Ollama LLM Integration

**User Story:** As a system operator, I want the backend to use the locally-hosted Ollama server, so that all inference runs on-premises without sending customer data to external services.

#### Acceptance Criteria

1. WHEN the `OLLAMA_HOST` environment variable is not set, THE Ollama_Client SHALL default to sending inference requests to `192.168.1.88` on port `11434` using the model `llama3.2`; WHEN `OLLAMA_HOST`, `OLLAMA_PORT`, or `OLLAMA_MODEL` are set, THE Ollama_Client SHALL use those values instead.
2. WHEN constructing an inference request, THE Ollama_Client SHALL include a system prompt that explicitly states the assistant's role as a customer service agent and instructs it to answer only questions related to the company's products and services.
3. WHEN constructing an inference request, THE Ollama_Client SHALL include the retrieved RAG context chunks ahead of the customer message in the prompt.
4. WHEN constructing an inference request, THE Ollama_Client SHALL include the Conversation_History up to the configured maximum number of prior Turns, defaulting to 10 Turns when not configured.
5. IF the combined token count of the system prompt, RAG context, Conversation_History, and user message would exceed the Context_Window of the `llama3.2` model, THEN THE Ollama_Client SHALL remove the oldest Turn from the Conversation_History one Turn at a time until the combined token count fits within the Context_Window.
6. IF the Ollama_Server returns a non-200 HTTP status, THEN THE Ollama_Client SHALL propagate the error to the API_Gateway with the original status code and error message.
7. IF the Ollama_Server is unreachable at the TCP connection level, THEN THE Ollama_Client SHALL return a connection-failure error to the API_Gateway distinct from an HTTP-level error.
8. WHEN the API request includes a `stream: true` flag, THE Ollama_Client SHALL request a streaming response from the Ollama_Server and forward each token chunk to the API_Gateway as it is received; WHEN the `stream` flag is absent or false, THE Ollama_Client SHALL request a non-streaming response and return the complete reply.
9. THE Ollama_Client SHALL apply a request timeout defaulting to 30 seconds, configurable via the `OLLAMA_TIMEOUT` environment variable.

### Requirement 4: RAG Pipeline — Query-Time Retrieval

**User Story:** As a customer, I want the chatbot to answer questions based on the company's knowledge base, so that responses are accurate and relevant to my specific situation.

#### Acceptance Criteria

1. WHEN a customer message is received, THE RAG_Pipeline SHALL convert the message into a vector embedding using the Embedding_Service before querying the Vector_DB.
2. IF the Embedding_Service fails to produce an embedding for the customer message, THEN THE RAG_Pipeline SHALL proceed with the LLM call without injecting context and SHALL include a flag in the API response indicating that retrieval was skipped due to an embedding error.
3. WHEN querying the Vector_DB, THE RAG_Pipeline SHALL retrieve the top-K most semantically similar Chunks to the customer message, where K is configurable and defaults to 5.
4. WHEN retrieved Chunks are available, THE RAG_Pipeline SHALL format them as a numbered context block and prepend the block to the LLM prompt.
5. IF the Vector_DB returns no results with a cosine similarity score above the configured similarity threshold (default 0.5), THEN THE RAG_Pipeline SHALL proceed with the LLM call without injecting context and SHALL include a flag in the API response indicating no knowledge base context was found.
6. WHEN retrieved Chunks are included in the context block, THE RAG_Pipeline SHALL include the source document name and chunk index for each Chunk in the context block.
7. WHEN the Vector_DB contains up to 100,000 Chunks and there is a single concurrent retrieval request, THE RAG_Pipeline SHALL complete retrieval within 2 seconds.

### Requirement 5: Document Ingestion Pipeline

**User Story:** As a knowledge base administrator, I want to ingest customer service documents into the vector database, so that the chatbot can retrieve relevant information when answering questions.

#### Acceptance Criteria

1. THE Ingestion_Pipeline SHALL accept input documents in PDF, plain text (`.txt`), and Markdown (`.md`) formats.
2. WHEN processing a document, THE Ingestion_Pipeline SHALL split the document into Chunks of configurable size between 64 and 2048 tokens (default 512 tokens) with configurable overlap between 0 and 50% of chunk size (default 50 tokens).
3. WHERE embedding generation fails for a Chunk, THE Ingestion_Pipeline SHALL store the Chunk text and source metadata (source document name, chunk index, raw chunk text) in the Vector_DB without an embedding and log a warning identifying the Chunk.
4. WHEN a Chunk is produced and embedding generation succeeds, THE Ingestion_Pipeline SHALL store the embedding together with the Chunk text and source metadata (source document name, chunk index, raw chunk text) in the Vector_DB.
5. IF a document with the same source path already exists in the Vector_DB, THEN THE Ingestion_Pipeline SHALL delete the existing Chunks for that document before re-ingesting, preventing duplicate entries.
6. WHEN the Ingestion_Pipeline finishes processing all files in the input directory, THE Ingestion_Pipeline SHALL log to stdout the total number of documents processed, total Chunks created, and total errors encountered.
7. THE Ingestion_Pipeline SHALL be executable as a standalone CLI command that accepts a directory path as input and processes all supported documents within that directory recursively.
8. WHEN an unsupported file format is encountered, THE Ingestion_Pipeline SHALL log a warning identifying the file and skip it without halting the overall ingestion process.
9. IF any error other than an unsupported file format occurs (such as a permission error, corrupted file, or Vector_DB network failure), THEN THE Ingestion_Pipeline SHALL log the error identifying the affected file, mark that file as failed in the summary report, and halt processing of all remaining files.
10. WHEN the Ingestion_Pipeline finishes or halts, THE Ingestion_Pipeline SHALL write a summary report to stdout listing each processed file, the number of Chunks generated, and the ingestion status (success, skipped, or error).
11. IF the Vector_DB is unavailable when the Ingestion_Pipeline attempts to store a Chunk, THEN THE Ingestion_Pipeline SHALL treat the failure as a network error per criterion 9 and halt.

---

### Requirement 6: Vector Database

**User Story:** As a system operator, I want a vector database to store and retrieve document embeddings, so that the RAG pipeline can perform fast semantic search over the knowledge base.

#### Acceptance Criteria

1. THE Vector_DB SHALL store document Chunks as vector embeddings with associated metadata including source document name, chunk index, and raw chunk text.
2. WHEN queried with a vector embedding and a value K greater than 0, THE Vector_DB SHALL return the K nearest neighbours by cosine similarity within 500 milliseconds for a knowledge base of up to 100,000 Chunks.
3. WHEN K exceeds the number of stored Chunks, THE Vector_DB SHALL return all available Chunks.
4. IF K is less than or equal to 0, THEN THE Vector_DB SHALL return an error response indicating an invalid K value.
5. THE Vector_DB SHALL support filtered queries that restrict results to a specific source document name or document category, where document category is a metadata field stored alongside each Chunk; IF a filtered query matches no Chunks, THEN THE Vector_DB SHALL return an empty result set.
6. THE Vector_DB SHALL persist stored embeddings to disk so that data survives container restarts.
7. THE Vector_DB SHALL expose a Health_Endpoint that returns HTTP 200 when the database is operational and HTTP 503 when it is not, so that the API_Gateway can poll it to verify connectivity.
8. THE Vector_DB SHALL expose a Metrics_Endpoint in Prometheus exposition format that the monitoring stack can scrape.

---

### Requirement 7: Session Management

**User Story:** As a customer, I want my conversation to maintain context across multiple messages, so that the chatbot understands follow-up questions without me repeating myself.

#### Acceptance Criteria

1. WHEN a chat request arrives with no session ID, THE API_Gateway SHALL generate a new UUID v4 session ID, create a new Session, and return the session ID in the response.
2. WHEN a chat request includes a session ID that exists in the server-side store and has not expired, THE API_Gateway SHALL retrieve the stored Conversation_History for that session and include it in the LLM prompt.
3. THE API_Gateway SHALL store Conversation_History server-side for the duration of the session, up to a configurable maximum number of Turns between 1 and 100 (default 20).
4. WHEN the Conversation_History reaches the maximum number of Turns, THE API_Gateway SHALL remove the oldest Turn before appending the new Turn.
5. IF a chat request includes a session ID that does not exist in the server-side store or whose session has expired, THEN THE API_Gateway SHALL create a new Session using that same session ID, proceed as a new conversation, and include a flag in the response indicating that prior history is unavailable.
6. THE API_Gateway SHALL expire sessions after a configurable idle timeout between 1 and 1440 minutes (default 30 minutes) from the time of the last request on that session.
7. WHEN a session is expired and a subsequent request arrives with that session ID, THE API_Gateway SHALL treat it as per criterion 5 and include the history-unavailable flag in the response.

### Requirement 8: Containerisation

**User Story:** As a DevOps engineer, I want all services containerised with Docker, so that the system can be deployed consistently across environments.

#### Acceptance Criteria

1. THE Chat_UI SHALL be packaged as a Docker image that serves the compiled Angular application via an Nginx web server listening on port 80 inside the container.
2. THE API_Gateway SHALL be packaged as a Docker image that exposes port 8000 inside the container and accepts all environment-specific configuration via environment variables.
3. THE Ingestion_Pipeline SHALL be packaged as a Docker image that can be run as a one-off container with a host directory mounted to `/documents` inside the container as the input document source.
4. THE Vector_DB SHALL be deployed using its official Docker image with a named volume mounted at the Vector_DB's designated data directory to ensure data persistence across container restarts.
5. THE docker/ directory SHALL contain a `docker-compose.yml` file that defines all services, networks, and volumes required to run the complete system locally with a single `docker compose up` command.
6. WHEN the `docker-compose.yml` is started, THE Chat_UI SHALL be accessible at `http://localhost:4200` and the API_Gateway SHALL be accessible at `http://localhost:8000`.
7. THE docker-compose.yml SHALL define a dedicated bridge network so that inter-service communication uses service names as hostnames.
8. THE Ingestion_Pipeline service SHALL be defined under a separate Docker Compose profile (e.g., `ingestion`) so that it does not start as part of the default `docker compose up` invocation.

---

### Requirement 9: Kubernetes Deployment

**User Story:** As a DevOps engineer, I want Kubernetes manifests for all services, so that the system can be deployed to a production-grade cluster.

#### Acceptance Criteria

1. THE k8s/ directory SHALL contain Kubernetes manifests for Deployment, Service, and ConfigMap resources for the Chat_UI, API_Gateway, and Ingestion_Pipeline (as a Kubernetes Job), and a StatefulSet, Service, and ConfigMap for the Vector_DB.
2. THE API_Gateway Deployment SHALL define a liveness probe and a readiness probe, both using `GET /api/v1/health`, with `initialDelaySeconds: 10`, `periodSeconds: 15`, and `failureThreshold: 3`.
3. THE Chat_UI Deployment SHALL define a liveness probe and a readiness probe, both using an HTTP GET on the Nginx root path (`/`), with `initialDelaySeconds: 5`, `periodSeconds: 10`, and `failureThreshold: 3`.
4. THE Vector_DB SHALL be deployed as a Kubernetes StatefulSet with a PersistentVolumeClaim of at least 10Gi to ensure data durability across pod restarts.
5. THE k8s/ directory SHALL contain a Kubernetes Secret manifest template with placeholder values for all sensitive configuration such as API keys and database credentials.
6. THE API_Gateway Deployment SHALL specify CPU resource requests so that the Horizontal Pod Autoscaler can calculate CPU utilisation.
7. WHERE a Horizontal Pod Autoscaler is configured, THE API_Gateway SHALL scale between a minimum of 1 and a maximum of 5 replicas based on CPU utilisation exceeding 70% of the requested CPU.

---

### Requirement 10: Monitoring and Observability

**User Story:** As a system operator, I want metrics and logs collected centrally, so that I can monitor system health, diagnose issues, and track chatbot usage.

#### Acceptance Criteria

1. THE API_Gateway SHALL expose the following Prometheus metrics at the `GET /metrics` endpoint: total request count labelled by HTTP status code, request duration histogram labelled by endpoint, active session count gauge, and RAG retrieval latency histogram.
2. THE monitoring/ directory SHALL contain a `prometheus.yml` configuration file that scrapes the API_Gateway `GET /metrics` endpoint and the Vector_DB Metrics_Endpoint at a 15-second interval.
3. THE monitoring/ directory SHALL contain an `alert_rules.yml` file referenced by `prometheus.yml` that defines the alerting rules for `HighErrorRate` and `HighLatency`.
4. THE monitoring/ directory SHALL contain a Grafana provisioning configuration that pre-loads a dashboard displaying: request rate, error rate, p50/p95/p99 response latency, active sessions, and RAG retrieval latency.
5. IF the proportion of HTTP 5xx responses among all API_Gateway responses exceeds 5% over a 5-minute window with at least 10 requests in that window, THEN the Prometheus alerting rules SHALL fire an alert named `HighErrorRate`.
6. IF the p95 response latency of the `POST /api/v1/chat` endpoint exceeds 10 seconds over a 5-minute window with at least 10 requests in that window, THEN the Prometheus alerting rules SHALL fire an alert named `HighLatency`.
7. THE API_Gateway SHALL emit structured JSON logs to stdout containing at minimum the fields defined in Requirement 2 criterion 10 (session ID, request duration in milliseconds, HTTP status code), so that a log aggregator can collect and index them.
8. THE monitoring/ directory SHALL contain a `docker-compose.monitoring.yml` file that adds Prometheus, Grafana, and a log aggregator to the local development environment.

---

### Requirement 11: Configuration and Security

**User Story:** As a system operator, I want all sensitive configuration externalised and the system hardened against common threats, so that the deployment is secure and maintainable.

#### Acceptance Criteria

1. THE API_Gateway SHALL read all environment-specific configuration (Ollama host, Ollama port, model name, Vector_DB URL, session timeout, CORS origin) from environment variables, with no hard-coded values in source code.
2. WHEN customer-supplied input is received, THE API_Gateway SHALL strip or escape characters that could alter the structure of the LLM prompt (including prompt-injection patterns such as "Ignore previous instructions") before including the input in any LLM request.
3. IF the sanitised input still contains a detected prompt-injection pattern after stripping, THEN THE API_Gateway SHALL reject the request with an HTTP 400 response and a JSON error body indicating the input was rejected for security reasons.
4. THE Chat_UI SHALL enforce a maximum message length of 2,000 characters, display a character count indicator to the customer, and block submission when the limit is reached.
5. THE API_Gateway SHALL enforce a maximum request body size of 32 KB.
6. IF a request body exceeds 32 KB, THEN THE API_Gateway SHALL return HTTP 413 with a JSON error body.
7. THE API_Gateway SHALL apply rate limiting of 60 requests per minute per IP address; IF the limit is exceeded, THEN THE API_Gateway SHALL return HTTP 429 with a `Retry-After` header indicating when the client may retry.
8. WHERE TLS termination is required, THE API_Gateway SHALL support configuration of TLS certificates via environment variables pointing to certificate and key file paths.
9. THE docker-compose.yml SHALL not include any secrets or credentials in plain text; all secrets SHALL be supplied via a `.env` file that is excluded from version control.
